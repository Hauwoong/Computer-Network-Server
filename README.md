# Computer-Network-Server

소켓(Socket) 기반으로 **HTTP/1.1 서버와 클라이언트를 직접 구현**한 컴퓨터 네트워크 과제입니다.
웹 프레임워크나 HTTP 라이브러리 없이 **Windows 소켓(Winsock2) TCP 통신**만으로 HTTP 요청을 파싱하고
응답을 조립하며, 데이터는 **SQLite**에 저장해 회원(users) 자원에 대한 **CRUD(생성·조회·수정·삭제)** 를 제공합니다.

이 과제의 핵심 목표는 단순히 요청에 응답하는 것을 넘어 **"실제 서버처럼 동작하게"** 만드는 것입니다.
즉, 오류 응답을 미리 정해 두고 무조건 돌려주는 것이 아니라 **서버의 실제 상태(DB 처리 결과·파일 존재
여부 등)를 확인한 뒤 그 상태에 맞는 HTTP 상태 코드**를 반환하도록 구현했습니다. 예를 들어 "없는 회원을
수정" 요청이 오면, 미리 만들어 둔 실패 응답을 던지는 것이 아니라 실제로 `UPDATE`를 시도하고
**변경된 행이 0개임을 확인한 뒤** `404`를 돌려줍니다.

## 1. 개발 환경 / 기술 스택

| 항목 | 내용 |
|------|------|
| 언어 | C++ (C++17 표준) |
| 컴파일러 | MinGW-w64 UCRT g++ 16 (POSIX threads 지원) |
| 네트워크 API | Windows Socket API (Winsock2, `ws2_32` 라이브러리) |
| 데이터베이스 | SQLite3 (단일 소스 amalgamation `sqlite3.c`) |
| 프로토콜 | HTTP/1.1 (지속 연결 keep-alive, Content-Length 프레이밍) |
| 동시성 | `std::thread` + `std::mutex` (연결별 스레드) |
| 포트 | 8080 |
| 운영체제 | Windows 11 |

라이브러리 의존성을 최소화하여 **HTTP 파싱·응답 생성·라우팅을 전부 직접 구현**했습니다.
외부에서 가져온 것은 데이터 저장을 위한 SQLite뿐이며, 네트워크는 표준 소켓 API만 사용했습니다.

## 2. 전체 구조와 동작 개요

프로그램은 **서버(`server.cpp`)** 와 **클라이언트(`client.cpp`)** 두 개의 독립 실행 파일로 구성됩니다.

```
   [클라이언트]                         [서버 : 포트 8080]
   client.exe                            server.exe
      │                                     │
      │  ── TCP 연결(connect) ──────────▶   accept()
      │                                     └─ 연결마다 스레드 생성(handleClient)
      │                                            │
      │  ── HTTP 요청(GET/POST/PUT/DELETE) ─▶  readOneRequest() 로 한 요청 읽기
      │                                            │
      │                                       buildResponse() 로 라우팅·처리
      │                                            │  (DB 접근은 mutex로 보호)
      │                                            ▼
      │                                        SQLite (data.db)
      │  ◀──────── HTTP 응답 ───────────────  sendAll() 로 전송
      │                                            │
      │  (같은 연결로 요청 반복 = keep-alive)      (연결 유지)
```

**흐름 요약**
1. 서버는 시작 시 DB와 테이블을 준비하고 8080 포트에서 연결을 기다립니다.
2. 클라이언트가 연결하면 서버는 **그 연결 전용 스레드**를 만들어 처리를 맡기고, 곧바로 다음 연결을 받습니다.
3. 한 연결에서 **여러 요청**을 주고받으며(keep-alive), 각 요청은 실제 서버/DB 상태를 확인해 응답합니다.
4. 클라이언트가 연결을 닫거나 종료 요청을 보내면 해당 스레드가 정리됩니다.

## 3. 서버 구현 상세

멀티스레드 HTTP/1.1 서버입니다. 아래는 주요 함수와 그 역할, 그리고 동작 방식입니다.

### 3-1. `main()` — 초기화와 연결 수락

1. **`WSAStartup`** 으로 Winsock 사용을 시작합니다.
2. **`sqlite3_open("data.db")`** 로 DB 파일을 열고,
   `CREATE TABLE IF NOT EXISTS users (id INTEGER PRIMARY KEY, data TEXT);` 로 테이블을 준비합니다.
   (`IF NOT EXISTS`이므로 기존 데이터는 보존됩니다 → 서버를 껐다 켜도 데이터 유지)
3. `socket` → `bind`(0.0.0.0:8080) → `listen` 으로 서버 소켓을 준비합니다.
4. **무한 루프**에서 `accept()`로 연결을 받고, 각 연결을 새 스레드에 넘깁니다.

```cpp
while (1) {
    SOCKET clientSock = accept(listenSock, nullptr, nullptr);
    if (clientSock == INVALID_SOCKET) continue;
    try {
        thread(handleClient, clientSock).detach();  // 이 연결은 새 스레드가 전담
    } catch (const exception& e) {                  // 스레드 생성 실패해도
        closesocket(clientSock);                     // 서버는 죽지 않고 연결만 거절
    }
}
```

메인 스레드는 연결을 넘기고 즉시 다음 `accept`로 돌아가므로, **한 클라이언트가 느려도 다른 클라이언트가
막히지 않습니다.**

### 3-2. `handleClient(SOCKET)` — 연결 하나를 전담하는 스레드

한 연결에서 요청이 오는 동안 반복 처리하는 **keep-alive 루프**입니다.

```
[handleClient 루프]
  ├ readOneRequest() 로 요청 하나를 정확히 읽음  (없으면=연결 종료 → 루프 탈출)
  ├ 첫 줄에서 method / path / version 파싱
  ├ 본문(body) 추출
  ├ POST /shutdown 이면 → 응답 후 서버 종료
  ├ { dbMutex 잠금 } response = buildResponse(...)   ← DB 접근 구간만 잠금
  ├ sendAll() 로 응답 전송
  └ 'Connection: close' 이면 루프 탈출
  (루프 끝: closesocket 으로 이 연결만 닫음)
```

DB를 다루는 `buildResponse` 호출만 **`dbMutex`로 잠가** 여러 스레드가 동시에 DB를 건드리지 않게 합니다.
반면 느린 작업인 `recv`/`send`(네트워크 대기)는 잠금 밖에 있어 **병렬로 처리**됩니다.

### 3-3. `readOneRequest(SOCKET, buf)` — HTTP 메시지 프레이밍

지속 연결(keep-alive)에서는 "연결이 끊길 때까지 읽기"가 통하지 않습니다(연결이 안 끊기니까요).
그래서 **한 요청의 경계를 정확히 찾아** 하나씩 잘라내야 합니다.

1. 헤더의 끝(`\r\n\r\n`)이 나올 때까지 `recv`로 받습니다.
2. 헤더에서 **`Content-Length`** 값을 읽어 본문 길이를 파악합니다.
3. `헤더 + 본문(Content-Length 바이트)` 만큼 다 받을 때까지 더 받습니다.
4. **요청 하나만** 잘라 반환하고, 남은 바이트(다음 요청 조각)는 버퍼에 남겨 둡니다.

또한 `Content-Length`가 **1MB를 초과하거나 음수**이면, 본문을 받으려 무한 대기하지 않고
즉시 `413 Payload Too Large`를 보내 연결을 끊습니다(악의적 요청으로 인한 서버 정지를 방지).

### 3-4. `buildResponse(method, path, body)` — 라우팅과 응답 생성

메서드와 경로를 보고 알맞은 HTTP 응답 문자열을 통째로 만들어 반환합니다.
라우팅은 **구체적인 경로를 먼저, 포괄적인 경로를 나중에** 검사합니다(순서가 매우 중요).

| 메서드 | 경로 | 동작 | 성공 | 실패(실제 상태 기반) |
|--------|------|------|------|----------------------|
| GET | `/` , `/index.html` 등 | 정적 파일(.html) 서빙 | 200 | 없으면 404 / `.html` 아니면 403 / `..` 400 |
| GET | `/users` | 회원 전체 목록 조회 | 200 | DB 오류 500 |
| GET | `/users/{id}` | 특정 회원 조회 | 200 | 없으면 404 / 잘못된 id 400 / DB 오류 500 |
| POST | `/users` | 회원 생성 | 201 (+`Location`) | 빈 본문 400 / DB 오류 500 |
| PUT | `/users/{id}` | 회원 수정 | 200 | 없으면 404 / 빈 본문·잘못된 id 400 / DB 오류 500 |
| DELETE | `/users/{id}` | 회원 삭제 | 200 | 없으면 404 / 잘못된 id 400 / DB 오류 500 |
| POST | `/shutdown` | 서버 종료(제어용) | 200 후 종료 | — |
| (그 외 메서드) | — | 미지원 메서드 | — | 405 (`Allow` 헤더 포함) |
| (그 외 경로) | — | 없는 경로 | — | 404 |

#### "실제 상태 기반" 오류 처리 (이 과제의 핵심)

오류 응답을 하드코딩하지 않고 **실제 서버 상태를 확인**해서 도출합니다.

- **404 Not Found**
  - PUT/DELETE: `UPDATE`/`DELETE` 실행 후 **`sqlite3_changes()`가 0**이면(=바뀐 행이 없음=대상이 실제로 없음) 404.
  - GET 단건: `sqlite3_step()`이 행을 주지 않으면(SQLITE_DONE) 404.
  - 파일: `ifstream`이 파일을 실제로 못 열면(`is_open()==false`) 404.
- **500 Internal Server Error**
  - `sqlite3_prepare_v2`/`sqlite3_step`의 반환 코드를 검사해 **실제 DB 오류**일 때, `sqlite3_errmsg`가 알려준 실제 오류 메시지를 담아 500.
- **400 Bad Request**
  - 회원 번호가 숫자가 아니거나 자릿수가 너무 클 때(정수 오버플로 방지), 생성/수정 본문이 비었을 때.
- **403 / 413**
  - `.html`이 아닌 파일 요청은 403, 과대한 본문은 413.

이렇게 하면 **오류 상황을 실제로 재현**했을 때 서버가 그 상태에 맞게 반응합니다.

### 3-5. 보조 함수

- **`formatData(data)`**: 저장 형식 `name=박준웅&age=10`을 사람이 읽기 좋은
  `name : 박준웅  age : 10`으로 변환합니다(`=` → ` : `, `&` → 공백). 저장은 원본을 유지하고 표시만 바꿉니다.
- **`sendAll(SOCKET, data)`**: `send()`는 한 번에 요청한 바이트를 다 못 보낼 수 있으므로,
  **전부 보낼 때까지 반복**해 응답이 잘리지 않게 합니다.

### 3-6. 동시성·보안 요약

- **멀티스레드**: 연결마다 스레드 → 동시 접속 처리, 느린 클라이언트가 전체를 막지 않음.
  전역 `sqlite3* db`는 `dbMutex`, 콘솔 로그는 `coutMutex`로 보호(데드락 없음).
- **SQL 인젝션 차단**: 모든 사용자 입력은 문자열을 이어붙이지 않고 **Prepared Statement + `sqlite3_bind_*`** 로 안전하게 처리.
- **정보 유출 차단**: 정적 파일은 `.html`만 서빙(소스코드·DB 파일 노출 방지).
- **DoS 방지**: 본문 크기 상한(1MB), 잘못된 입력에도 스레드가 죽지 않도록 방어.

## 4. 클라이언트 구현 상세

서버에 **한 번 연결(지속 연결)** 하고 메뉴를 반복하는 대화형 클라이언트입니다.
사용자가 HTTP 경로나 폼(form) 형식을 몰라도 되도록 입력을 최대한 단순화했습니다.

### 4-1. 주요 함수

- **`sendAll`**: 서버와 동일하게, 요청을 끝까지 전송.
- **`readOneResponse(SOCKET, buf)`**: 서버의 `readOneRequest`와 대칭. 응답 헤더의 `Content-Length`를 읽어
  **응답 하나를 정확히** 잘라 읽습니다(지속 연결에서 응답 경계를 구분하는 핵심).
- **`getBody(msg)`**: HTTP 메시지에서 본문(빈 줄 뒤)만 추출.

### 4-2. 동작 흐름

매 메뉴 반복마다 먼저 `GET /users`를 보내 **현재 회원 목록을 자동으로 보여주고**, 사용자가 무엇을 할지
선택하게 합니다. 이렇게 하면 "몇 번 회원이 있는지" 보고 나서 조회/수정/삭제할 수 있습니다.

```
========================================
 현재 등록된 회원 목록
========================================
1번: name : park  age : 10
========================================

어떤 요청을 보낼까요?
 1) 조회 (GET)
 2) 생성 (POST)
 3) 수정 (PUT)
 4) 삭제 (DELETE)
 9) 클라이언트만 종료 (서버는 계속 실행)
 0) 전체 종료 (서버까지 종료)
 번호 선택:
```

### 4-3. 입력 단순화

| 메뉴 | 사용자 입력 | 클라이언트가 만드는 요청 |
|------|-------------|--------------------------|
| 2) 생성 | 이름, 나이 | `POST /users`, 본문 `name=...&age=...` |
| 1) 조회 | 회원 번호(0=전체) | `GET /users` 또는 `GET /users/{번호}` |
| 3) 수정 | 회원 번호 → 이름, 나이 | `PUT /users/{번호}`, 본문 `name=...&age=...` |
| 4) 삭제 | 회원 번호 | `DELETE /users/{번호}` |

- 사용자는 `name=kim&age=20` 같은 형식을 몰라도 되고, 클라이언트가 자동으로 조립합니다.
- 생성(POST)은 항상 `/users`에 만들며 **id는 서버가 자동 부여**합니다(REST의 일반적 설계).

### 4-4. 없는 회원 사전 확인 (PUT/DELETE)

수정·삭제 시 번호를 입력하면, 클라이언트가 **먼저 `GET /users/{번호}`로 존재를 확인**합니다.
없으면 이름·나이 입력 같은 헛수고 없이 바로 안내하고 메뉴로 돌아갑니다.

```
회원 번호를 입력하세요: 999

[안내] 999번은 없는 회원번호입니다. 다시 선택해주세요.
```

### 4-5. 한글 지원

`SetConsoleOutputCP(CP_UTF8)`(출력)와 `SetConsoleCP(CP_UTF8)`(입력)로 콘솔 입출력을 모두 UTF-8로
맞춰, 한글 이름을 입력해도 깨지지 않고 저장·표시됩니다.

### 4-6. 종료 옵션

- **9) 클라이언트만 종료**: 연결만 닫고 종료. 서버는 계속 실행되어 다른 클라이언트/브라우저가 사용 가능.
- **0) 전체 종료**: `POST /shutdown`을 보내 서버까지 종료.

## 5. HTTP 요청·응답 예시

실제로 오가는 HTTP 메시지의 예시입니다.

### 회원 생성 (성공)
요청
```
POST /users HTTP/1.1
Host: 127.0.0.1:8080
Content-Type: application/x-www-form-urlencoded
Content-Length: 16

name=park&age=10
```
응답
```
HTTP/1.1 201 Created
Location: /users/1
Content-Type: text/plain; charset=utf-8
Content-Length: 50

1번 회원 생성됨! 데이터: name=park&age=10
```
→ 생성된 자원의 위치를 `Location` 헤더로 알려주고, 서버가 부여한 id(1)를 안내합니다.

### 없는 회원 조회 (실제 상태 기반 404)
```
GET /users/999 HTTP/1.1        →      HTTP/1.1 404 Not Found

                                      999번 회원이 없습니다.
```
→ DB를 실제 조회한 결과 행이 없어서 404가 나갑니다(하드코딩된 실패가 아님).

### 미지원 메서드 (405)
```
PATCH /users HTTP/1.1          →      HTTP/1.1 405 Method Not Allowed
                                      Allow: GET, POST, PUT, DELETE

                                      지원하지 않는 메서드입니다: PATCH
```

### 잘못된 회원 번호 (400)
```
GET /users/abc HTTP/1.1        →      HTTP/1.1 400 Bad Request

                                      잘못된 회원 번호입니다: abc
```

### 접근 불가 파일 (403)
```
GET /server.cpp HTTP/1.1       →      HTTP/1.1 403 Forbidden

                                      허용되지 않는 파일 형식입니다. (.html만 접근 가능)
```

### 회원 수정 (성공)
요청
```
PUT /users/1 HTTP/1.1
Host: 127.0.0.1:8080
Content-Type: application/x-www-form-urlencoded
Content-Length: 16

name=lee&age=25
```
응답
```
HTTP/1.1 200 OK
Content-Type: text/plain; charset=utf-8
Content-Length: 46

1번 회원 수정됨! 새 데이터: name=lee&age=25
```

### 없는 회원 수정 (실제 상태 기반 404)
```
PUT /users/999 HTTP/1.1        →      HTTP/1.1 404 Not Found

                                      999번 회원이 없어서 수정할 수 없습니다.
```
→ 서버는 실제로 `UPDATE ... WHERE id = 999`를 실행하고, **변경된 행 수가 0**임을 확인한 뒤 404를 보냅니다.

### 회원 삭제 (성공)
```
DELETE /users/1 HTTP/1.1       →      HTTP/1.1 200 OK

                                      1번 회원 삭제됨!
```

### 빈 본문 생성 (400)
```
POST /users HTTP/1.1           →      HTTP/1.1 400 Bad Request
Content-Length: 0
                                      생성할 데이터(본문)가 없습니다.
```

### 과대한 본문 (413) — DoS 방어
```
POST /users HTTP/1.1           →      HTTP/1.1 413 Payload Too Large
Content-Length: 999999999             Connection: close
(본문 안 보냄)
                                      요청 본문이 너무 큽니다. (최대 1MB)
```
→ 본문을 받으려 무한 대기하지 않고 즉시 413을 보내고 연결을 닫아 서버가 멈추지 않습니다.

## 6. 빌드 및 실행 방법

### 6-1. 빌드

SQLite 소스(`sqlite3.c`)는 **C 컴파일러(gcc)** 로 따로 컴파일해야 합니다.
`sqlite3.c`는 C로 작성돼 있어, C++ 컴파일러로 빌드하면 `new` 같은 C++ 예약어와 충돌해 에러가 납니다.
그래서 SQLite는 gcc로 목적파일(`.o`)을 만든 뒤, C++ 코드와 링크합니다.

```bash
gcc -c sqlite3.c -o sqlite3.o                     # SQLite 컴파일 (한 번만)
g++ server.cpp sqlite3.o -o server.exe -lws2_32   # 서버 빌드
g++ client.cpp -o client.exe -lws2_32             # 클라이언트 빌드
```

`-lws2_32`는 Winsock 라이브러리를 링크하는 옵션입니다(빠뜨리면 `socket`·`bind` 등에서 링크 에러).

### 6-2. 실행

서버는 계속 떠 있는 프로그램이므로 **터미널 두 개**(또는 서버 + 브라우저)를 사용합니다.

```powershell
# 터미널 1 - 서버
.\server.exe
# 출력: DB 준비 완료 (data.db)
#       서버 시작! 포트 8080에서 연결 대기중...

# 터미널 2 - 클라이언트
.\client.exe
```

## 7. 사용 방법 / 실행 시나리오

### 7-1. 클라이언트로 사용 (GET/POST/PUT/DELETE 전부)

메뉴에서 번호를 고르고 안내에 따라 입력합니다. 대표 시나리오:

1. `2`(생성) → 이름 `park`, 나이 `10` → `1번 회원 생성됨`
2. `1`(조회) → 번호 `1` → `name : park  age : 10`
3. `3`(수정) → 번호 `1` → 이름 `kim`, 나이 `25` → `1번 회원 수정됨`
4. `4`(삭제) → 번호 `1` → `1번 회원 삭제됨`
5. 없는 번호(예: `999`)로 수정/삭제 → "없는 회원번호입니다" 안내 후 메뉴 복귀

실제 화면 예(생성):
```
----------------------------------------
 보낸 요청
----------------------------------------
POST /users HTTP/1.1
Host: 127.0.0.1:8080
Content-Type: application/x-www-form-urlencoded
Content-Length: 16

name=park&age=10
----------------------------------------

----------------------------------------
 받은 응답
----------------------------------------
HTTP/1.1 201 Created
Location: /users/1
Content-Type: text/plain; charset=utf-8
Content-Length: 50

1번 회원 생성됨! 데이터: name=park&age=10
----------------------------------------
```

### 7-2. 브라우저로 사용

- **조회(GET)** 는 주소창에서 바로 됩니다:
  - `http://localhost:8080/` (메인 페이지)
  - `http://localhost:8080/users` (회원 목록)
  - `http://localhost:8080/users/1` (1번 회원)
- **생성·수정·삭제**는 주소창으로 불가능하므로(주소창은 GET만) **F12 → Console**에서 `fetch`로 보냅니다:
  ```js
  // 생성
  fetch('/users', {method:'POST',
    headers:{'Content-Type':'application/x-www-form-urlencoded'},
    body:'name=kim&age=20'}).then(r=>r.text()).then(console.log)

  // 수정
  fetch('/users/1', {method:'PUT',
    headers:{'Content-Type':'application/x-www-form-urlencoded'},
    body:'name=lee&age=30'}).then(r=>r.text()).then(console.log)

  // 삭제
  fetch('/users/1', {method:'DELETE'}).then(r=>r.text()).then(console.log)
  ```
- **Network 탭**을 열어두면 각 요청의 상태 코드와 응답 헤더(`Location`, `Allow` 등)를 확인할 수 있습니다.

### 7-3. Wireshark로 패킷 확인
- 같은 PC 통신(127.0.0.1)은 루프백으로 흐르므로 **Npcap Loopback Adapter**로 캡처합니다.
- 표시 필터: `tcp.port == 8080` 또는 `http`. keep-alive이므로 `Follow TCP Stream`으로 한 연결의 여러 요청/응답을 한눈에 볼 수 있습니다.

### 7-4. 종료 / 데이터 유지
- 종료: 클라이언트 메뉴 `0`(전체) 또는 서버 터미널에서 `Ctrl + C`.
- 데이터는 `data.db`에 저장되어 **서버를 껐다 켜도 유지**됩니다. 초기화하려면 서버를 끈 뒤 `data.db`를 삭제하세요.

## 8. 개발 중 겪은 문제와 해결

개발하면서 마주친 모든 문제를 "문제 상황 → 원인 → 해결" 순으로 정리했습니다.
성격별로 묶었으며, 번호는 실제 개발/수정 순서를 반영합니다.
(⭐는 이번 과제의 핵심 요구인 "실제 상태 기반 동작"과 직결되는 항목)

### 8-1. 개발 환경 · 빌드

**1. 콘솔에서 한글이 깨져서 출력됨**
- 문제: 한글 출력이 `???`나 깨진 글자로 나옴.
- 원인: Windows 콘솔 기본 인코딩과 소스 파일 인코딩(UTF-8)이 다름.
- 해결: 소스를 UTF-8로 저장하고, 시작 시 `SetConsoleOutputCP(CP_UTF8);`로 콘솔 출력 코드페이지를 UTF-8로 맞춤.

**2. 소켓 함수에서 링크 에러(undefined reference)**
- 문제: `socket`, `bind`, `WSAStartup`에서 `undefined reference`로 컴파일 실패.
- 원인: Winsock 함수는 `ws2_32` 라이브러리에 있는데 링크하지 않음.
- 해결: 컴파일 시 `-lws2_32` 옵션 추가.

**3. SQLite 소스(sqlite3.c)를 g++로 컴파일하니 에러 폭탄**
- 문제: `g++`로 `sqlite3.c`를 함께 컴파일하니 `expected unqualified-id before 'new'` 등 다수 에러.
- 원인: `sqlite3.c`는 C 언어라, C에서 평범한 식별자 `new` 등이 C++에서는 예약어라 충돌.
- 해결: SQLite는 C 컴파일러(`gcc -c sqlite3.c -o sqlite3.o`)로 따로 컴파일 후 C++ 코드와 링크.

**7. 포트가 이미 사용 중이거나 DB 파일이 잠김**
- 문제: 재실행 시 `bind 실패`가 나거나 `data.db`가 "사용 중(busy)"으로 지워지지 않음.
- 원인: 이전 서버 프로세스가 종료되지 않고 남아 포트·DB를 점유.
- 해결: `taskkill /F /IM server.exe`로 잔여 프로세스를 종료 후 재실행.

**8. localhost(127.0.0.1) 통신이 Wireshark에 안 잡힘**
- 문제: 이더넷/와이파이 인터페이스로 캡처해도 HTTP 패킷이 안 보임.
- 원인: 같은 PC 통신(127.0.0.1)은 랜카드를 안 거치고 루프백(loopback)으로만 흐름.
- 해결: Npcap의 **Loopback Adapter**로 캡처하고 필터 `http` 또는 `tcp.port == 8080` 적용.

### 8-2. HTTP 파싱 · 라우팅 · 프로토콜

**9. 클라이언트 요청 형식 오류로 서버 파싱이 깨짐**
- 문제: 서버가 요청을 제대로 해석 못 함.
- 원인: 시작줄 조립 시 경로와 버전 사이 공백이 빠져 `GET /index.htmlHTTP/1.1`처럼 붙음.
- 해결: `method + " " + path + " HTTP/1.1\r\n"`로 각 요소 사이 공백을 정확히 넣음.

**5. `GET /users`가 404로 처리되던 문제**
- 문제: DB 조회용 `GET /users`를 추가했는데도 404가 반환됨.
- 원인: "그 외 모든 GET → 404" 규칙이 `GET /users`보다 위에 있어 먼저 걸림.
- 해결: 구체적 경로(`/users`)를 포괄적 규칙보다 **위**에 배치(라우팅은 구체적 → 일반적 순서).

**6. HEAD 메서드를 과제 범위에서 제외**
- 문제: 초기엔 HEAD도 처리했으나 과제 요구는 GET·POST·PUT(·DELETE) 구성이었음.
- 원인: HEAD는 본문 없이 헤더만 회신하는 특수 메서드로 학습 범위와 안 맞음.
- 해결: 서버·클라이언트에서 HEAD 관련 코드를 모두 제거하고 메뉴를 재정리.

**19. 매 요청마다 연결을 끊던 문제 → HTTP/1.1 keep-alive 전환 ⭐**
- 문제: 요청 1건 처리 후 곧바로 연결을 끊어, 한 소켓에 한 요청만 가능(HTTP/1.1 기본과 다름).
- 원인: 응답의 끝을 "연결이 끊기는 것"으로 판단했기 때문.
- 해결: **Content-Length 기반 메시지 프레이밍**(`readOneRequest`/`readOneResponse`) 구현.
  헤더 끝(`\r\n\r\n`)까지 읽고 Content-Length만큼 본문을 읽어 메시지 1개를 정확히 잘라내고,
  남은 바이트는 버퍼에 남겨 다음 메시지로 처리. 한 연결에서 여러 요청을 주고받게 됨.
  (검증: curl 요청 3건에서 `num_connects` 1→0→0으로 연결 재사용 확인)

**20. DELETE 라우팅을 fallback(405) 뒤에 두어 도달 불가였던 문제**
- 문제: `DELETE /users/{id}`를 추가했는데도 항상 405가 반환됨.
- 원인: DELETE 블록이 "미지원 메서드 → 405" fallback **뒤**에 있어 fallback이 먼저 return(도달 불가). 405 조건에도 DELETE가 빠져 있었음.
- 해결: DELETE 케이스를 fallback보다 위(PUT 다음)로 이동, 405 조건에 DELETE 추가, `Allow` 헤더도 갱신.

### 8-3. 실제 상태 기반 오류 처리 (이 과제의 핵심)

**10. PUT 요청이 실제 데이터를 수정하지 않던 문제**
- 문제: PUT이 받은 데이터를 돌려주기만 하고 DB는 그대로였음.
- 원인: PUT 케이스가 응답만 만들고 DB 작업을 안 함.
- 해결: 실제로 `UPDATE users SET data=? WHERE id=?`를 실행하도록 변경.

**11. 오류 응답(NOK)을 실제 서버 상태와 무관하게 회신하던 문제 ⭐**
- 문제: PUT이 대상 존재 여부와 상관없이 무조건 `200 수정됨`을 반환(없는 회원도 성공으로 응답).
- 원인: `sqlite3_step()` 후 실제 변경 건수를 확인하지 않고 응답을 하드코딩.
- 해결: `UPDATE` 실행 뒤 **`sqlite3_changes(db)`로 실제 변경 행 수를 확인** → 0이면 404, 아니면 200.
  (검증: 빈 DB에 `PUT /users/1` → 실제 404, 회원을 넣은 뒤 → 200)

**13. DB 작업 실패를 감지하지 못하고 무조건 성공으로 회신하던 문제 ⭐**
- 문제: `prepare`/`step`이 실패해도 결과 코드를 확인 안 해, 내부 오류가 나도 정상 응답이 나갈 수 있었음.
- 원인: SQLite는 실패 시 예외가 아니라 에러 코드를 반환하는데, 그 반환 코드를 검사하지 않음.
- 해결: 각 호출의 반환 코드를 검사하고 실패 시 `sqlite3_errmsg(db)`의 실제 오류 메시지를 담아 500.
  PUT은 **① step 실패(500) → ② changes==0(404) → ③ 성공(200)** 순으로 분기(순서 중요).

**14. GET의 404도 실제 서버 상태를 보고 판단하도록 변경 ⭐**
- 문제: `GET /index.html`이 코드에 박아둔 문자열을 반환하고, 그 외 경로는 "목록에 없다"는 이유만으로 404.
- 원인: 정적 자원을 실제 파일로 다루지 않고 라우팅 실패를 곧 404로 취급.
- 해결: 정적 파일은 실제로 열어(`ifstream`) 있으면 200/없으면 404, `..`는 400.
  회원 단건은 `SELECT ... WHERE id=?`로 조회해 행이 있으면 200/없으면 404.

**15. 없는 파일인데 200이 반환되던 문제 (`is_open` 필요)**
- 문제: 없는 파일 요청에도 `200 OK`(빈 본문)가 나감.
- 원인: 사용 컴파일러(MinGW-w64 UCRT)에서 없는 파일로 `ifstream` 생성 시 `if(file)`이 참으로 평가됨(`fail()`은 false, `is_open()`은 false).
- 해결: 파일 존재는 **`file.is_open()`으로 판정** → 없는 파일은 정상적으로 404.

**18. 지원하지 않는 요청에 가짜 200을 회신하던 문제 ⭐**
- 문제: 아무 케이스에도 안 걸린 요청(예: `DELETE /users`, `POST /wrong`)에 무조건 `200 OK`(빈 본문)를 반환.
- 원인: 함수 끝 임시 fallback이 `200 OK`로 하드코딩돼 있었음.
- 해결: 미지원 메서드는 `405`(+`Allow` 헤더), 지원 메서드지만 경로 없음은 `404`.
  함수의 모든 경로가 반드시 return하도록 정리(이전엔 return 누락 경고/미정의 동작 위험).

### 8-4. CRUD 기능 보강

**12. 특정 번호(id)의 회원만 수정할 수 없던 문제**
- 문제: PUT 경로가 `/users/1`로 고정돼 1번만 수정 가능.
- 원인: 경로를 문자열 비교로만 매칭하고 id를 동적으로 해석하지 않음.
- 해결: `path.rfind("/users/",0)==0`으로 검사하고 뒤 문자열을 잘라 id로 변환(`substr(7)`)해 바인딩.

**21. POST가 본문 검증·생성 위치 안내 없이 부실하던 문제**
- 문제: 빈 본문도 그대로 저장하며 201을 반환하고, 생성된 자원의 id를 알려주지 않음.
- 해결: 빈 본문은 400으로 거부, `sqlite3_last_insert_rowid(db)`로 새 id를 얻어 `Location: /users/{id}` 헤더와 번호를 응답에 포함.

**26. PUT이 빈 본문도 성공(200)으로 처리해 데이터를 지우던 문제**
- 문제: PUT이 빈 본문에도 `data`를 빈 값으로 UPDATE하고 200을 반환(기존 데이터가 지워짐).
- 원인: PUT에 본문 유효성 검사가 없었음.
- 해결: POST와 동일하게 빈 본문이면 400. (검증: 빈 PUT 후에도 기존 데이터 보존)

**34. 회원 목록·조회 결과를 보기 좋게 표시**
- 문제: 저장 원본 `name=박준웅&age=10` 형태로 그대로 출력돼 읽기 불편.
- 해결: 서버에 `formatData()`를 추가해 `=`→` : `, `&`→공백으로 변환 → `name : 박준웅  age : 10`(저장은 원본 유지, 표시만 변환).

### 8-5. 안정성 · 보안

**4. SQL 인젝션 위험**
- 문제: 사용자 입력을 SQL 문자열에 직접 이어붙이면 따옴표·악의적 구문으로 조작 가능.
- 해결: **Prepared Statement**(`?` 자리표시자 + `sqlite3_bind_*`)로 값을 안전하게 바인딩.

**16. 숫자가 아닌 id로 요청하면 서버가 죽던 문제 (stoi 예외)**
- 문제: `GET /users/abc` 등에서 `stoi`가 `std::invalid_argument` 예외 → 서버 종료.
- 해결: `stoi` 전에 숫자로만 이뤄졌는지 검사(`find_first_not_of`)하고, 아니면 400(사전 검증으로 예외 회피).

**17. 숫자지만 너무 큰 id로 요청하면 서버가 죽던 문제 (stoi 오버플로)**
- 문제: `GET /users/99999999999999999`처럼 int 범위 초과 숫자에서 `std::out_of_range` 예외 → 종료.
- 원인: 숫자 여부만 보고 자릿수(값 크기)는 안 봤음.
- 해결: 유효성 검사에 자릿수 제한(≤9)을 추가. GET·PUT·DELETE 모든 핸들러에 동일 적용.

**22. 파일 서빙이 작업 폴더의 모든 파일을 노출하던 문제 (정보 유출) ⭐**
- 문제: `GET /server.cpp`(소스), `GET /data.db`(DB 전체) 등 아무 파일이나 다운로드 가능.
- 원인: 서빙 가능한 파일 종류를 제한하지 않음.
- 해결: **`.html`로 끝나는 파일만** 허용하고 그 외는 403 Forbidden(화이트리스트).

**23. 악의적으로 큰 Content-Length로 서버를 멈출 수 있던 문제 (DoS) ⭐**
- 문제: `Content-Length: 999999999`만 선언하고 본문을 안 보내면 서버가 무한 대기(hang).
- 검토: `recv` 타임아웃도 있으나 idle keep-alive 연결을 끊는 부작용이 있음.
- 해결: 본문 상한(1MB) 초과/음수면 본문을 받지 않고 즉시 413 후 연결 종료(부작용 0, atoi 오버플로/음수도 방어).

**24. `send()`가 데이터를 한 번에 다 못 보내 응답이 잘릴 수 있던 문제**
- 문제: `send()`를 한 번만 호출하고 반환값을 안 봐, 소켓 버퍼가 차면 일부만 전송될 수 있었음.
- 해결: 전부 보낼 때까지 반복하는 `sendAll()` 헬퍼를 만들어 서버·클라이언트의 모든 전송에 적용.

**25. 한 번에 한 연결만 처리하던 문제 → 멀티 스레드 ⭐**
- 문제: 단일 스레드라 한 클라이언트가 느리면(idle/느린 전송) 다른 클라이언트가 모두 대기.
- 해결: 연결마다 `std::thread`를 만들어(`handleClient`) 위임, main은 즉시 다음 `accept`로 복귀.
  공유 자원 `db`는 `dbMutex`로 `buildResponse` 구간만 잠금(느린 recv/send는 잠금 밖 → 병렬).
  스레드 함수 끝의 `closesocket` 필수(소켓 누수 방지).
  (검증: 동시 POST 20건 정확히 20건 저장, 느린 연결 점유 중 정상 요청 약 1.3ms 응답)

**28. 멀티 스레드에서 콘솔 로그가 뒤섞이던 문제**
- 문제: 여러 스레드가 동시에 `cout` 출력 시 줄이 섞임(동작 무해, 가독성 저하).
- 해결: 로그 전용 `coutMutex`로 각 로그 출력을 `lock_guard`로 감싸 한 줄씩 완결(데드락 없음).

**29. 스레드 생성 실패 시 서버 전체가 죽던 문제**
- 문제: `thread(...).detach()`가 자원 고갈 시 `std::system_error`를 던지면, 예외가 전파돼 서버 전체 종료.
- 해결: 스레드 생성을 `try/catch`로 감싸 실패 시 그 연결만 닫고 서버는 계속 동작.

### 8-6. 사용성(UX) · 클라이언트

**27. 루트 경로("/") 요청이 403을 반환하던 문제**
- 문제: `GET /`가 403을 반환(파일명이 빈 문자열이 돼 `.html` 화이트리스트에 걸림).
- 해결: 파일명이 비면 `index.html`로 대체(기본 문서 관례) → `GET /`는 메인 페이지 200.

**30. 클라이언트에서 서버를 종료하는 제어 명령 추가**
- 필요: 서버 터미널에서 `Ctrl+C` 대신 클라이언트 메뉴로 종료.
- 구현: `POST /shutdown`을 받으면 서버가 200 응답 후 `exit(0)`로 종료. (인증 없는 원격 종료는 실무에선 위험 → localhost 데모용)

**31. POST인데도 클라이언트가 경로를 물어 혼란스럽던 문제**
- 문제: POST는 항상 `/users`에만 만드는데 경로를 묻는 게 무의미하고 헷갈림.
- 해결: POST는 경로 프롬프트 생략(`/users` 자동), 데이터만 입력. id는 서버가 부여.

**32. 한글 이름 입력이 깨지던 문제 (콘솔 입력 인코딩)**
- 문제: 한글 이름 입력 시 `name : ???`처럼 깨져 저장·표시됨.
- 원인: 출력만 UTF-8로 맞추고 입력 코드페이지는 안 맞춰 `cin`이 CP949로 읽음. (서버는 바이트를 그대로 저장하므로 서버 문제 아님)
- 해결: 클라이언트에 `SetConsoleCP(CP_UTF8)` 추가(입력도 UTF-8). 이미 깨져 저장된 데이터는 `data.db` 삭제로 초기화.

**33. 경로·폼 형식을 직접 입력해야 하던 불편 → 입력 친화화**
- 문제: `/users/3` 경로와 `name=kim&age=20` 형식을 사용자가 직접 입력해야 했음.
- 해결: 메서드별로 필요한 값만 받도록 단순화 — POST=이름·나이, GET=번호(0=전체), PUT/DELETE=번호(+PUT은 이름·나이). 클라이언트가 요청을 조립.

**35. PUT/DELETE 시 없는 번호를 미리 걸러내기 (사전 확인)**
- 문제: PUT은 번호→이름·나이까지 입력한 뒤에야 404가 나 헛수고.
- 해결: 번호 입력 직후 클라이언트가 `GET /users/{번호}`로 존재를 먼저 확인 → 없으면 안내 후 메뉴 복귀. (서버는 여전히 실제 요청에도 404를 정확히 반환 → 일관성 유지)

**36. 콘솔 출력 정리 및 종료 옵션 명확화**
- 문제: 구분선 길이가 제각각이고 여백이 없어 지저분. 종료 옵션의 차이도 불명확.
- 해결: 폭 40칸 `BAR`/`DASH` 상수로 구분선 통일·여백 추가, 목록을 `name : X  age : Y`로 표시.
  종료를 `9) 클라이언트만 종료(서버 유지)` / `0) 전체 종료(서버까지)`로 분리·명확화.

### 8-7. 알려진 한계 (과제 범위상 남겨둔 것)

- **connection-per-thread**: 연결마다 스레드라 수천~수만 동시접속엔 부적합. 대규모엔 스레드풀이나 `select`/`epoll`/IOCP가 필요(스레드 생성 실패 자체는 #29로 방어).
- **단순화한 헤더 파싱**: `Content-Length`를 버퍼 전체 문자열 검색으로 찾아, 경로에 `Content-Length:`를 심은 악의적 요청은 본문 길이를 잘못 계산할 수 있음(정상 요청엔 무영향). 정식 파서는 헤더를 줄 단위로 파싱해야 함.
- **클라이언트 입력 견고성**: 메뉴에 숫자가 아닌 값을 넣으면 `cin`이 실패 상태가 됨(대개 깔끔히 종료되나 방어적 처리는 없음).
- **DB 잠금 직렬화**: `buildResponse` 전체를 잠가 DB 작업이 직렬화됨(정확성 우선). 읽기 병렬성이 필요하면 스레드별 연결이나 읽기/쓰기 잠금 분리가 필요.
- **slowloris형 공격**: 본문을 아주 느리게 보내며 연결을 오래 점유하는 공격은 크기 상한으로는 못 막음(본문 읽기 전용 타임아웃 필요). 멀티스레드라 서버 전체가 멈추지는 않음.

## 10. 정리 / 배운 점

- **HTTP는 결국 텍스트 규약**임을 소켓 수준에서 직접 다루며 이해했습니다. 요청 라인·헤더·빈 줄·본문의
  구조와 `Content-Length`로 메시지 경계를 잡는 프레이밍이 왜 필요한지 keep-alive를 구현하며 체감했습니다.
- **"실제 서버처럼"** 이라는 요구를 통해, 오류를 미리 정해 두고 던지는 것과 **실제 상태를 확인해 도출**하는
  것의 차이를 명확히 알게 되었습니다(`sqlite3_changes`, `errmsg`, 파일 존재 확인 등).
- **동시성**(스레드+뮤텍스), **보안**(SQL 인젝션·파일 유출·DoS 방어), **인코딩**(UTF-8) 등 실제 서버가
  신경 써야 하는 요소들을 작은 규모로나마 직접 구현하며 배웠습니다.
- 남은 한계(대규모 동시접속용 스레드풀, 정교한 헤더 파서 등)는 과제 범위를 넘어서므로
  "알려진 한계"로 문서화했습니다.
