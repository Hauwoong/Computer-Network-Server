# 개발 중 겪은 문제와 해결 과정 (Troubleshooting)

> README에 넣을 초안입니다. 실제 개발하면서 마주친 문제들을 "문제 상황 → 원인 → 해결" 순서로 정리했습니다.

---

## 1. 콘솔에서 한글이 깨져서 출력됨

- **문제 상황:** 서버/클라이언트에서 한글(`서버 시작!` 등)을 출력하니 `???`나 깨진 글자로 나옴.
- **원인:** Windows 콘솔의 기본 문자 인코딩과 소스 파일의 인코딩(UTF-8)이 달라서.
- **해결:** ① 소스 파일을 **UTF-8**로 저장하고, ② 프로그램 시작 시 `SetConsoleOutputCP(CP_UTF8);`를 호출해 콘솔 출력 코드페이지를 UTF-8로 맞춤.

---

## 2. 컴파일 시 소켓 함수에서 링크 에러 (undefined reference)

- **문제 상황:** `socket`, `bind`, `WSAStartup` 등에서 `undefined reference` 에러가 나며 컴파일 실패.
- **원인:** Winsock 함수들은 별도 라이브러리(`ws2_32`)에 들어 있는데, 링크 단계에서 이 라이브러리를 같이 묶어주지 않음.
- **해결:** 컴파일 명령에 `-lws2_32` 옵션을 추가.
  ```
  g++ server.cpp -o server.exe -lws2_32
  ```

---

## 3. SQLite 소스(sqlite3.c)를 g++로 컴파일하니 에러 폭탄

- **문제 상황:** `g++ server.cpp sqlite3.c ...`로 한 번에 컴파일하니 `redefinition`, `expected unqualified-id before 'new'` 등 수십 개 에러 발생.
- **원인:** `sqlite3.c`는 **C 언어**로 작성됨. 그런데 g++(C++ 컴파일러)로 컴파일하면, C에서는 평범한 식별자인 `new` 등이 C++에서는 **예약어**라 충돌함.
- **해결:** SQLite는 **C 컴파일러(gcc)로 따로** 컴파일해 목적파일(`.o`)로 만든 뒤, C++ 코드와 링크.
  ```
  gcc -c sqlite3.c -o sqlite3.o            # C로 컴파일 (부품 생성)
  g++ server.cpp sqlite3.o -o server.exe -lws2_32   # C++과 링크
  ```

## 4. SQL 인젝션 위험 (보안 문제)

- **문제 상황:** POST로 받은 데이터를 SQL 문자열에 그대로 이어붙이면, 입력에 따옴표(`'`)나 악의적 구문이 들어올 때 SQL이 깨지거나 조작될 수 있음.
- **원인:** 사용자 입력을 SQL 명령에 직접 합치는 방식(문자열 연결)의 근본적 취약점.
- **해결:** **Prepared Statement(준비된 문장)** 사용. SQL에 `?` 자리표시자를 두고, 실제 값은 `sqlite3_bind_text()`로 안전하게 바인딩.
  ```cpp
  sqlite3_prepare_v2(db, "INSERT INTO users (data) VALUES (?);", -1, &stmt, nullptr);
  sqlite3_bind_text(stmt, 1, body.c_str(), -1, SQLITE_TRANSIENT);
  ```

## 5. GET /users 요청이 404로 처리되는 문제

- **문제 상황:** DB 조회용 `GET /users`를 추가했는데도 404 Not Found가 반환됨.
- **원인:** 요청 라우팅을 위에서부터 순서대로 `if`로 검사하는데, "그 외 모든 GET → 404" 규칙이 `GET /users`보다 **위에** 있어서 먼저 걸려버림.
- **해결:** 구체적인 경로(`/users`) 케이스를 **포괄적인 404 케이스보다 위쪽**에 배치. (라우팅은 구체적인 것 → 일반적인 것 순서)

## 6. HEAD 메서드를 과제 범위에서 제외

- **문제 상황:** 초기에는 HEAD 요청도 처리했으나, 과제 요구사항은 GET·PUT·POST 세 가지 메서드로 구성하는 것이었음.
- **원인:** HEAD는 본문 없이 헤더만 회신하는 특수 메서드로, 이번 과제의 학습 범위(자원의 조회·생성·수정)와 맞지 않음.
- **해결:** 서버의 HEAD 라우팅 케이스와 클라이언트 메뉴의 HEAD 선택지를 **모두 제거**하고, 메뉴 번호를 다시 정리해 GET·POST·PUT 중심으로 재구성.

## 7. 포트가 이미 사용 중이거나 DB 파일이 잠김

- **문제 상황:** 서버를 다시 실행하니 `bind 실패`가 나거나, `data.db` 파일이 "사용 중(busy)"이라 지워지지 않음.
- **원인:** 이전에 실행한 서버 프로세스가 **종료되지 않고 남아** 8080 포트와 `data.db`를 계속 점유.
- **해결:** 남아 있는 서버 프로세스를 종료한 뒤 재실행.
  ```
  taskkill /F /IM server.exe
  ```

## 8. localhost(127.0.0.1) 통신이 Wireshark에 안 잡힘

- **문제 상황:** Wireshark에서 이더넷/와이파이 인터페이스로 캡처해도 우리 HTTP 패킷이 안 보임.
- **원인:** 같은 PC 안에서의 통신(127.0.0.1)은 일반 랜카드를 거치지 않고 **루프백(loopback)** 으로만 흐름.
- **해결:** Npcap의 **`Adapter for loopback traffic capture`** 인터페이스로 캡처하고, 표시 필터 `http` 또는 `tcp.port == 8080` 적용.

## 9. 클라이언트 요청 형식 오류로 서버 파싱이 깨짐

- **문제 상황:** 클라이언트가 보낸 요청을 서버가 제대로 해석하지 못함.
- **원인:** 요청 시작줄을 조립할 때 경로와 버전 사이 **공백이 빠져** `GET /index.htmlHTTP/1.1`처럼 붙어버림. 서버는 공백으로 `메서드/경로/버전`을 나눠 읽기 때문에 파싱이 깨짐.
- **해결:** 시작줄 조립 시 각 요소 사이에 공백을 정확히 넣음: `method + " " + path + " HTTP/1.1\r\n"`.

## 10. PUT 요청이 실제 데이터를 수정하지 않던 문제

- **문제 상황:** PUT은 "수정" 메서드인데, 받은 데이터를 돌려주기만 하고 DB는 그대로였음.
- **원인:** 초기 구현에서 PUT 케이스가 응답만 생성하고 데이터베이스에는 아무 작업을 하지 않음.
- **해결:** PUT 케이스에서 실제로 `UPDATE users SET data = ? WHERE id = ?;`를 실행하도록 변경. 이후 POST(저장) → PUT(수정) → GET(확인)으로 데이터가 실제로 바뀌는 것을 검증.

## 11. 오류 응답(NOK)을 실제 서버 상태와 무관하게 회신하던 문제 ⭐

- **문제 상황:** PUT 케이스가 대상 회원의 존재 여부와 관계없이 **무조건 `200 OK`("수정됨")** 를 회신함. 즉 실제로는 없는 회원을 수정하려 해도 성공했다고 응답. 반대로 오류 응답을 넣더라도 서버 상태를 확인하지 않고 미리 만들어 둔 NOK를 바로 돌려주는 것은 실제 서버 동작이 아님.
- **원인:** `sqlite3_step()` 실행 후 **실제로 몇 건이 변경됐는지 확인하지 않고** 응답을 하드코딩함.
- **해결:** `UPDATE` 실행 뒤 **`sqlite3_changes(db)`로 실제 변경된 행 수를 확인**하여, 상태에 근거해 응답을 분기.
  - 변경 행 수 `== 0` → 해당 회원이 실제로 존재하지 않음 → **`404 Not Found`** (실제 상황 기반의 NOK)
  - 변경 행 수 `>= 1` → 실제로 수정됨 → **`200 OK`**
  ```cpp
  sqlite3_step(stmt);
  sqlite3_finalize(stmt);
  int changed = sqlite3_changes(db);   // 실제 DB 상태 확인
  if (changed == 0) { /* 404 Not Found */ }
  else              { /* 200 OK */ }
  ```
  - 검증: DB가 비어 있을 때 `PUT /users/1` → 실제로 404. POST로 회원을 넣은 뒤 같은 요청 → 200. (오류 상황을 실제로 재현해 응답이 상태에 따라 달라짐을 확인)

## 12. 특정 번호(id)의 회원만 수정할 수 없던 문제

- **문제 상황:** PUT 경로가 `/users/1`로 고정되어 있어 1번 회원만 수정 가능. 2번·3번 등 다른 회원은 수정 불가.
- **원인:** 경로를 문자열 비교(`path == "/users/1"`)로만 매칭하고, 경로에 담긴 id를 동적으로 해석하지 않음.
- **해결:** 경로가 `/users/`로 시작하는지 `path.rfind("/users/", 0) == 0`으로 검사하고, 그 뒤 문자열을 잘라 id로 변환해 SQL에 바인딩.
  ```cpp
  string idStr = path.substr(7);   // "/users/" 는 7글자 -> 그 뒤가 id
  int id = stoi(idStr);            // "/users/3" -> 3
  sqlite3_bind_int(stmt, 2, id);   // WHERE id = ? 에 바인딩
  ```
  - 주의: `"/users/"`는 7글자이므로 `substr(7)`이어야 함. `substr(6)`이면 `"/3"`이 잘려 `stoi`가 `std::invalid_argument` 예외를 던지고 서버가 종료됨.

## 13. DB 작업 실패를 감지하지 못하고 무조건 성공으로 회신하던 문제 ⭐

- **문제 상황:** GET·POST·PUT의 DB 작업(`prepare`/`step`)이 실패해도 결과 코드를 확인하지 않아, 내부적으로 오류가 나도 클라이언트에는 정상 응답이 나갈 수 있었음.
- **원인:** `sqlite3_prepare_v2()`와 `sqlite3_step()`의 **반환 코드를 검사하지 않음.** SQLite는 실패 시 예외를 던지지 않고 에러 코드를 반환하므로, 검사하지 않으면 오류가 조용히 무시됨.
- **해결:** 각 DB 호출의 반환 코드를 실제로 확인하고, 실패 시 `sqlite3_errmsg(db)`가 알려주는 **실제 DB 오류 메시지**를 담아 `500 Internal Server Error`로 회신.
  - `sqlite3_prepare_v2(...) != SQLITE_OK` → SQL 준비 실패 → **500**
  - `sqlite3_step(...) != SQLITE_DONE` → 실행 실패 → **500**
  ```cpp
  if (sqlite3_step(stmt) != SQLITE_DONE) {
      string msg = "서버 DB 오류: ";
      msg += sqlite3_errmsg(db);   // DB가 실제로 알려준 오류
      sqlite3_finalize(stmt);
      /* 500 Internal Server Error 회신 */
  }
  ```
  - PUT에서는 검사 순서가 중요: **① step 실패(500) → ② `sqlite3_changes()==0`(404) → ③ 성공(200)** 순으로 분기. step이 실패한 상태에서 `changes()`를 신뢰하면 안 되므로 500을 먼저 걸러냄.
  - **실제 500 재현 방법:** `CREATE TABLE ...` 줄을 잠깐 주석 처리하거나 `users` 테이블이 없는 DB로 실행하면 `INSERT`/`UPDATE`가 "no such table: users" 오류를 실제로 반환 → 진짜 500 응답이 나가고 Wireshark에서도 확인됨. (시연 후 원복)

## 14. GET의 404도 실제 서버 상태를 보고 판단하도록 변경 ⭐

- **문제 상황:** GET 응답이 실제 자원 상태와 무관했음. `GET /index.html`은 코드에 박아둔 HTML 문자열을 반환했고, 그 외 경로는 "if 목록에 없다"는 이유만으로 무조건 404를 반환(자원의 실제 존재 여부를 확인하지 않음).
- **원인:** 정적 자원을 실제 파일로 다루지 않고, 라우팅 실패를 곧 404로 취급함.
- **해결:** 두 가지로 나눠 실제 상태 기반으로 응답을 도출.
  - **정적 파일:** 요청 경로를 실제 파일로 열어(`ifstream`) **존재하면 파일 내용으로 200, 없으면 404**. 경로에 `..`가 있으면 `400 Bad Request`로 차단.
  - **회원 단건:** `GET /users/{id}`는 `SELECT ... WHERE id = ?`로 조회해 **행이 있으면 200, 없으면 404**(`sqlite3_step()`의 `SQLITE_ROW`/`SQLITE_DONE`으로 판정).
- **라우팅 순서 주의:** 포괄적인 `if (method == "GET")`(파일 서빙) 블록은 **반드시 `GET /users`·`GET /users/{id}`보다 아래**에 둬야 함. 위에 두면 모든 GET을 먼저 가로채 DB 라우트가 동작하지 않음(구체적 경로 → 일반적 경로 순서).

## 15. 없는 파일인데 200이 반환되던 문제 (ifstream `is_open` 필요)

- **문제 상황:** 존재하지 않는 파일을 요청해도 `200 OK`(빈 본문)가 나감. `if (file)`로 열림 여부를 검사했는데 통과됨.
- **원인:** 사용 중인 컴파일러(MinGW-w64 UCRT)에서 없는 파일로 `ifstream`을 생성해도 `if (file)`(= `!fail()`)이 참으로 평가됨. 실제로 `is_open()`은 `false`인데 `fail()`은 `false`인 상태.
- **해결:** 파일 존재 여부는 **`file.is_open()`으로 판정.** `if (file)` → `if (file.is_open())`로 변경하니 없는 파일은 정상적으로 404가 반환됨.
  ```cpp
  ifstream file(filename, ios::binary);
  if (file.is_open()) { /* 200 + 파일 내용 */ }
  else                { /* 404 Not Found */ }
  ```

## 16. 숫자가 아닌 id로 요청하면 서버가 죽던 문제 (stoi 예외)

- **문제 상황:** `GET /users/abc`, `PUT /users/`처럼 id 자리에 숫자가 아닌 값이 오면 `stoi()`가 `std::invalid_argument` 예외를 던져 **서버 프로세스가 종료**됨.
- **원인:** 경로에서 잘라낸 문자열을 검증 없이 `stoi()`로 변환. 잘못된 요청 하나가 서버 전체를 내리는 상황(실제 서버라면 있어선 안 됨).
- **해결:** `stoi()` 전에 **문자열이 숫자로만 이루어졌는지 검사**하고, 아니면 `400 Bad Request`로 회신. try/catch 대신 사전 검증으로 예외 자체를 발생시키지 않음.
  ```cpp
  if (idStr.empty() || idStr.find_first_not_of("0123456789") != string::npos)
  {
      /* 400 Bad Request */
  }
  int id = stoi(idStr);   // 통과했으면 숫자임이 보장되어 안전
  ```
  - 검증: `GET /users/abc` → 400, `PUT /users/xyz` → 400, 그리고 **서버가 죽지 않고** 이후 요청도 정상 처리됨을 확인.

## 17. 숫자지만 너무 큰 id로 요청하면 서버가 죽던 문제 (stoi 오버플로)

- **문제 상황:** `GET /users/99999999999999999`처럼 **숫자로만 이뤄졌지만 int 범위를 넘는** id를 보내면 `stoi()`가 `std::out_of_range` 예외를 던져 서버가 종료됨. 숫자 검사(#16)는 통과하므로 놓치기 쉬움.
- **원인:** 숫자 여부만 검사하고 **자릿수(값의 크기)는 검사하지 않음.** `int`의 최대값은 약 21억(10자리)이라 그보다 큰 숫자열은 변환 시 넘침.
- **해결:** 유효성 검사에 **자릿수 제한**을 추가. 9자리(최대 999,999,999)면 `int` 범위 안에 안전.
  ```cpp
  if (idStr.empty()
      || idStr.find_first_not_of("0123456789") != string::npos
      || idStr.size() > 9)          // 오버플로 방지
  { /* 400 Bad Request */ }
  ```
  - GET·PUT 두 핸들러 모두에 동일하게 적용해야 함(한쪽만 고치면 다른 쪽에서 여전히 크래시).

## 18. 지원하지 않는 요청에 가짜 200을 회신하던 문제 ⭐

- **문제 상황:** 라우팅에서 아무 케이스에도 안 걸린 요청(예: `DELETE /users`, `POST /wrong`, id 없는 `PUT /users`)에 대해 마지막에 **무조건 `200 OK`(빈 본문)** 를 반환. 실제로는 처리하지 못했는데 성공인 것처럼 응답.
- **원인:** 함수 끝의 임시 fallback이 `return "HTTP/1.1 200 OK...";`로 하드코딩돼 있었음. 이는 "실제 상태와 무관한 OK 회신"으로 부적절.
- **해결:** 요청 종류에 맞는 실제 상태 코드로 분기.
  - 지원하지 않는 **메서드**(DELETE 등) → `405 Method Not Allowed` (+ `Allow: GET, POST, PUT` 헤더로 지원 목록 안내)
  - 지원하는 메서드지만 **경로가 없음** → `404 Not Found`
- **부수 효과:** 이 과정에서 함수의 모든 경로가 반드시 `return`하도록 정리(마지막 404를 무조건 반환). 이전에는 특정 경로에서 return이 빠져 `control reaches end of non-void function` 경고 + 미정의 동작 위험이 있었음.

## 19. 매 요청마다 연결을 끊던 문제 → HTTP/1.1 지속 연결(keep-alive)로 전환 ⭐

- **문제 상황:** 초기 구현은 요청 1건을 처리한 뒤 곧바로 `closesocket`으로 연결을 끊고(`Connection: close`), 다음 요청은 새 연결을 맺어야 했음. HTTP/1.1의 기본(지속 연결)과 다름.
- **원인:** 응답의 끝을 "연결이 끊기는 것"으로 판단했기 때문. 이러면 한 소켓으로 한 번밖에 요청을 못 보냄.
- **해결:** **Content-Length 기반 메시지 프레이밍**을 구현해 한 연결에서 여러 요청·응답을 주고받도록 변경.
  - 헤더 끝(`\r\n\r\n`)까지 읽고 → `Content-Length`만큼 본문을 더 읽어 **메시지 1개를 정확히 잘라냄**(`readOneRequest`/`readOneResponse`). 나머지 바이트는 버퍼에 남겨 다음 메시지로 처리.
  - 서버: `accept` 후 안쪽 루프에서 요청을 반복 처리, 클라이언트가 연결을 닫거나 `Connection: close`를 보낼 때만 종료.
  - 클라이언트: 한 번 연결해 메뉴를 반복(요청 여러 번), 종료 시에만 `closesocket`.
  - 검증: curl로 요청 3건을 보내니 `num_connects`가 1→0→0(연결 재사용)으로 확인됨.
- **주의:** 지속 연결에서는 **응답 길이를 미리 알아야**(Content-Length) `recv`를 멈추고 소켓을 재사용할 수 있음. "recv를 EOF까지 읽기"는 close 방식에서만 통함.
- **참고:** 100 Continue(1xx 중간 응답)는 본문이 없어 이 프레이밍과 별도 처리가 필요하므로, keep-alive 전환 시 데모에서 제외함.

## 20. DELETE 라우팅을 fallback(405) 뒤에 두어 도달 불가였던 문제

- **문제 상황:** `DELETE /users/{id}` 케이스를 추가했는데도 항상 `405 Method Not Allowed`가 반환됨.
- **원인:** DELETE 처리 블록을 함수 끝의 "지원하지 않는 메서드 → 405" fallback **뒤**에 배치. fallback이 먼저 `return`하여 DELETE 케이스에 도달하지 못함(unreachable). 또한 405 판정 조건에 DELETE가 빠져 있어 지원 메서드인데도 405로 처리됨.
- **해결:** DELETE 케이스를 **fallback보다 위**(PUT 다음)로 이동하고, 405 조건을 `method != "GET" && ... && method != "DELETE"`로 수정. `Allow` 헤더도 `GET, POST, PUT, DELETE`로 갱신.
- 동작: DELETE는 PUT과 동일 패턴(경로에서 id 추출 → 검증 → `DELETE FROM users WHERE id=?` → `sqlite3_changes()`로 판정) → 실제 삭제 200 / 없으면 404 / 숫자오류 400 / DB오류 500.

## 21. POST가 본문 검증·생성 위치 안내 없이 부실하던 문제

- **문제 상황:** `POST /users`가 (1) 빈 본문도 그대로 빈 행으로 저장하며 201을 반환하고, (2) 생성된 자원의 id를 알려주지 않아 클라이언트가 무엇이 어디에 만들어졌는지 알 수 없었음.
- **원인:** 생성 성공/DB오류만 다루고, 요청 유효성 검사와 REST의 `201 Created` 표준(생성 위치 안내)을 구현하지 않음.
- **해결:**
  - **빈 본문 → `400 Bad Request`**: 생성할 데이터가 없으면 실제 상태 기반으로 거부.
  - **`sqlite3_last_insert_rowid(db)`** 로 방금 INSERT된 행의 id를 얻어, 응답에 **`Location: /users/{id}` 헤더**와 생성된 번호를 포함. → 클라이언트가 곧바로 `GET /users/{id}`로 확인 가능.
  ```cpp
  if (body.empty()) { /* 400 Bad Request */ }
  ...
  int newId = (int)sqlite3_last_insert_rowid(db);
  // 응답에 "Location: /users/" + to_string(newId) 헤더 추가
  ```

## 22. 파일 서빙이 작업 폴더의 모든 파일을 노출하던 문제 (정보 유출) ⭐

- **문제 상황:** 파일 서빙이 요청 경로의 파일을 그대로 열어주다 보니, `GET /server.cpp`로 **소스코드 전체**, `GET /data.db`로 **DB 파일 통째로**, `GET /sqlite3.h` 등 작업 폴더의 아무 파일이나 다운로드할 수 있었음. (`..` 상위 폴더 접근은 막았지만 폴더 내부 파일은 무방비)
- **원인:** 서빙 가능한 파일의 **종류를 제한하지 않음.** 존재하는 파일이면 무엇이든 200으로 반환.
- **해결:** **`.html` 확장자로 끝나는 파일만** 서빙하도록 화이트리스트 검사를 추가하고, 그 외에는 `403 Forbidden` 반환.
  ```cpp
  bool isHtml = filename.size() >= 5 && filename.substr(filename.size() - 5) == ".html";
  if (!isHtml) { /* 403 Forbidden */ }
  ```
  - 검증: `GET /server.cpp`·`/data.db`·`/sqlite3.h` → 403, `GET /index.html` → 200, `GET /nope.html` → 404(형식은 맞지만 없음). 소스·DB 유출 차단 확인.

## 23. 악의적으로 큰 Content-Length로 서버를 멈출 수 있던 문제 (DoS) ⭐

- **문제 상황:** 클라이언트가 `Content-Length: 999999999`만 선언하고 본문을 보내지 않으면, 서버가 그 바이트를 다 받으려 `recv`에서 **무한 대기(hang)**. 서버가 단일 스레드라 그동안 **모든 클라이언트가 멈춤**(DoS).
- **원인:** 선언된 본문 크기를 **검증 없이 그대로 신뢰**하고 다 받을 때까지 기다림.
- **검토한 대안:** `recv` 타임아웃(`SO_RCVTIMEO`)도 방법이지만, **keep-alive로 열어둔 idle 연결**(대화형 클라이언트가 메뉴에서 입력 대기 중)까지 끊어버리는 부작용이 있어 idle/본문 타임아웃을 분리해야 함.
- **해결(채택):** 본문 크기 **상한(1MB)** 검사. `Content-Length`가 상한 초과이거나 음수면 본문을 받지 않고 **즉시 `413 Payload Too Large`** 반환 후 연결 종료(`Connection: close`).
  ```cpp
  const int MAX_BODY = 1024 * 1024;   // 1MB
  if (contentLength < 0 || contentLength > MAX_BODY)
  { /* 413 Payload Too Large 보내고 return "" -> 연결 닫음 */ }
  ```
  - 부작용 0(idle keep-alive에 영향 없음), atoi 오버플로/음수도 함께 방어.
  - 검증: raw 소켓으로 `Content-Length: 999999999` 전송 → 413 즉시 응답, **직후 정상 요청도 200**(서버 생존). 정상 POST(1MB 이하)는 그대로 201.

## 24. send()가 데이터를 한 번에 다 못 보내 응답이 잘릴 수 있던 문제

- **문제 상황:** `send()`를 한 번만 호출하고 반환값을 확인하지 않음. `send`는 소켓 송신 버퍼가 차면 **요청한 바이트 중 일부만 보내고** 나머지는 보내지 않을 수 있어, 큰 응답(파일 등)이 잘려 전송될 수 있었음.
- **원인:** `send`의 반환값(실제 전송한 바이트 수)을 무시하고 1회 호출로 끝냄.
- **해결:** **다 보낼 때까지 반복하는 `sendAll()` 헬퍼**를 만들어 서버·클라이언트의 모든 전송을 교체.
  ```cpp
  bool sendAll(SOCKET sock, const string& data) {
      int totalSent = 0;
      while (totalSent < (int)data.size()) {
          int sent = send(sock, data.c_str() + totalSent, (int)data.size() - totalSent, 0);
          if (sent <= 0) return false;   // 전송 실패
          totalSent += sent;
      }
      return true;
  }
  ```
  - 서버 응답 전송·413 전송, 클라이언트 요청 전송에 모두 적용. 이제 응답/요청이 커도 잘리지 않고 끝까지 전송됨.

## 25. 한 번에 한 연결만 처리하던 문제 → 멀티 스레드로 동시 접속 처리 ⭐

- **문제 상황:** 단일 스레드 구조라 `accept` 후 그 연결을 끝까지 처리한 뒤에야 다음 연결을 받음. 한 클라이언트가 느리거나(keep-alive로 idle) 본문을 천천히 보내면 **다른 모든 클라이언트가 대기**.
- **원인:** main 스레드가 연결 처리와 수락을 모두 담당.
- **해결:** **연결마다 `std::thread`를 생성**해 처리를 위임하고 main은 즉시 다음 `accept`로 복귀.
  - 한 연결 처리 로직을 `handleClient(SOCKET)` 함수로 분리, `thread(handleClient, clientSock).detach();`로 실행.
  - **공유 자원(전역 `sqlite3* db`) 보호:** `std::mutex dbMutex`로 **`buildResponse` 구간만 잠금**(`lock_guard`). DB 작업은 한 번에 한 스레드만 → 데이터 레이스와 `sqlite3_last_insert_rowid` 경합 차단. 반면 느린 `recv`/`send`는 잠금 밖이라 병렬 유지 → 동시성 확보.
  - 스레드 함수 끝에서 **`closesocket(clientSock)` 필수**(빠뜨리면 연결마다 소켓 핸들 누수).
  - 검증: 동시 POST 20건 → 정확히 20건 저장(레이스 없음). 느린 클라이언트가 연결을 6초 점유하는 동안 정상 요청이 약 1.3ms에 응답(안 막힘).

## 26. PUT이 빈 본문도 성공(200)으로 처리해 데이터를 지우던 문제

- **문제 상황:** POST는 빈 본문을 `400`으로 거부하는데, PUT은 빈 본문이 와도 `data`를 빈 문자열로 UPDATE하고 **`200 수정됨`** 을 반환. 즉 데이터 없이 보낸 요청이 "성공"으로 나가고, 기존 회원 데이터가 빈 값으로 덮어써졌음.
- **원인:** PUT에 본문 유효성 검사가 없어, id만 맞으면 무조건 수정 실행.
- **해결:** PUT에도 POST와 **동일한 빈 본문 검사**를 추가 → 빈 본문이면 `400 Bad Request`.
  ```cpp
  if (body.empty()) { /* 400 Bad Request */ }
  ```
  - 검증: `PUT /users/1`(빈 본문) → 400, 이후에도 1번 데이터가 보존됨. 정상 본문은 그대로 200.

## 27. 루트 경로("/") 요청이 403을 반환하던 문제

- **문제 상황:** `GET /`(주소만 입력한 접속)가 `403 Forbidden`을 반환. 파일 서빙에서 경로의 `/`를 떼면 파일명이 빈 문자열이 되어 `.html` 화이트리스트에 걸림.
- **원인:** 루트 경로에 대한 기본 문서(index) 처리가 없었음.
- **해결:** 파일명이 비면 **`index.html`로 대체**(실무 웹서버의 기본 문서 관례).
  ```cpp
  string filename = path.substr(1);
  if (filename.empty()) filename = "index.html";
  ```
  - 검증: `GET /` → 200(index.html), `GET /index.html`·`GET /server.cpp`(403)·`GET /users` 등 기존 동작은 그대로.

## 28. 멀티 스레드에서 콘솔 로그가 뒤섞이던 문제

- **문제 상황:** 여러 스레드가 동시에 `cout`으로 로그를 찍으면 `[요청] ...` 등의 줄이 서로 섞여 읽기 어려웠음(동작에는 무해, 시연·디버깅 시 가독성 저하).
- **원인:** `cout` 삽입은 스레드 안전하지만, 여러 `<<` 출력이 스레드 사이에서 끼어들 수 있음.
- **해결:** 로그 전용 `std::mutex coutMutex`를 두고 각 로그 출력을 `lock_guard`로 감싸 한 번에 한 줄씩 완결되게 함. `dbMutex`와 중첩 잠금이 없어 데드락 위험도 없음.
- **부수 발견:** 이 작업 중 편집 과정에서 `handleClient` 끝의 `closesocket(clientSock)`가 사라져(소켓 누수 재발) 다시 추가함 → 회귀 방지를 위해 연결 종료 처리를 항상 확인.

## 29. 스레드 생성 실패 시 서버 전체가 죽던 문제

- **문제 상황:** `thread(handleClient, clientSock).detach();`는 스레드를 만들 수 없으면(자원 고갈 등) `std::system_error`를 던지는데, 이 예외가 `accept` 루프 밖으로 전파되면 **잡는 곳이 없어 서버 프로세스 전체가 종료**됨. 연결 폭주에 취약.
- **원인:** 스레드 생성에 예외 처리가 없음.
- **해결:** 스레드 생성을 `try/catch`로 감싸, 실패 시 경고 로그 후 **그 연결만 닫고(`closesocket`) 서버는 계속 동작**.
  ```cpp
  try { thread(handleClient, clientSock).detach(); }
  catch (const exception& e) { /* 경고 로그 + closesocket(clientSock) */ }
  ```
  - 검증: 동시 접속 15건 정상 처리, 크래시·누수 없음.

## 30. 클라이언트에서 서버를 종료하는 제어 명령 추가

- **필요:** 매번 서버 터미널에서 `Ctrl+C`로 끄는 대신, 클라이언트 메뉴로 서버와 클라이언트를 함께 종료하고 싶음.
- **구현:** 제어용 엔드포인트 `POST /shutdown` 추가.
  - 서버(`handleClient`): 이 요청을 받으면 `200 서버를 종료합니다.` 응답을 **먼저 보낸 뒤** `exit(0)`로 프로세스 전체를 종료(모든 스레드 함께). `buildResponse`로 넘기지 않고 handleClient에서 직접 처리.
  - 클라이언트: 메뉴 `9`번 선택 시 `POST /shutdown`을 보내고, 응답을 출력한 뒤 루프를 빠져나와 종료.
- **주의:** 인증 없는 원격 종료 엔드포인트는 실무에선 위험(아무나 서버를 끌 수 있음). 여기서는 **localhost 데모/편의용**으로만 사용.
- 검증: 클라이언트 `9` 선택 → 서버가 200 응답 후 프로세스 종료, 이후 요청은 연결 실패(서버 꺼짐) 확인.

## 31. POST인데도 클라이언트가 경로를 물어 혼란스럽던 UX 문제

- **문제 상황:** 클라이언트가 모든 메서드에 대해 "경로를 입력하세요"를 물음. 그런데 POST(생성)는 항상 `/users` 컬렉션에만 만들 수 있어(id는 서버가 부여), 경로를 묻는 것이 무의미하고 사용자가 `/users/5` 등을 입력하면 오히려 헷갈렸음.
- **원인:** 자원 컬렉션이 `/users` 하나라 POST 대상이 고정인데, 입력 흐름은 일반화돼 있었음.
- **해결:** POST일 때는 경로 프롬프트를 생략하고 `path = "/users"`로 자동 지정 + 안내 문구 출력. GET/PUT/DELETE는 id가 달라지므로 경로 입력을 유지.
  - 결과: POST는 "데이터만 입력"으로 단순해지고, 생성 후 서버가 부여한 id는 응답의 `Location`/메시지로 확인.

## 32. 한글 이름 입력이 깨지던 문제 (콘솔 입력 인코딩)

- **문제 상황:** 클라이언트에서 한글 이름을 입력하면 `name : ???` 처럼 깨져서 저장·표시됨.
- **원인:** 출력만 `SetConsoleOutputCP(CP_UTF8)`로 UTF-8로 맞추고 **입력 코드페이지는 안 맞춤**. 그래서 `cin`이 한글을 CP949 바이트로 읽어들여, UTF-8로 표시될 때 깨짐. (서버는 받은 바이트를 그대로 저장·반환하므로 서버 문제가 아님.)
- **해결:** 클라이언트에 **`SetConsoleCP(CP_UTF8)`**(입력도 UTF-8) 추가. 입력·출력 양방향 UTF-8이 되어 한글이 온전히 흐름.
  - 주의: 수정 전에 깨진 채 저장된 데이터는 그대로 남으므로, 초기화하려면 `data.db`를 삭제.

## 33. 클라이언트 입력을 사용자 친화적으로 개선 (경로·폼 형식 감추기)

- **문제 상황:** 모든 요청에서 `/users/3` 같은 **경로**와 `name=kim&age=20` 같은 **폼 형식**을 사용자가 직접 입력해야 했음. 형식을 모르면 쓰기 어렵고, POST는 어차피 `/users`뿐이라 경로를 묻는 것도 무의미(#31).
- **해결:** 메서드별로 필요한 값만 자연어로 입력받고 클라이언트가 요청을 조립.
  - **POST**: 경로 안 물음(→`/users` 고정), **이름·나이**만 입력 → `name=...&age=...` 자동 조립.
  - **GET**: **회원 번호**만 입력(0이면 전체 목록) → `/users` 또는 `/users/{번호}`.
  - **PUT/DELETE**: **회원 번호**만 입력 → `/users/{번호}` (+ PUT은 이름·나이).
  - 트레이드오프: 클라이언트로는 파일/임의 경로를 못 보냄(그건 브라우저 담당). 회원 API에 집중된 깔끔한 UX.

## 34. 회원 목록·조회 결과를 보기 좋게 표시

- **문제 상황:** 목록·단건 조회가 저장 원본인 `name=박준웅&age=10` 형태로 그대로 출력돼 읽기 불편.
- **해결:** 서버에 `formatData()` 헬퍼를 추가해 `=` → ` : `, `&` → 공백으로 변환 → **`name : 박준웅  age : 10`** 으로 표시. (저장은 원본 유지, 표시만 변환.)

---

## 35. PUT/DELETE 시 없는 번호를 미리 걸러내기 (클라이언트 사전 확인)

- **문제 상황:** PUT은 회원 번호 → 이름·나이까지 입력한 뒤에야 서버가 404를 돌려줌 → 헛수고. DELETE도 없는 번호에 요청을 보냄.
- **해결:** PUT·DELETE에서 번호 입력 직후 **클라이언트가 `GET /users/{번호}`로 존재를 먼저 확인**. 200이 아니면 "없는 회원번호입니다" 안내 후 `continue`로 메뉴 복귀(PUT은 이름·나이 입력도 생략).
  - 서버는 여전히 실제 PUT/DELETE 요청에도 404를 정확히 반환하므로, 서버의 상태 기반 처리와 일관됨.

## 36. 콘솔 출력 정리 및 종료 옵션 명확화

- **문제 상황:** 출력의 구분선 길이가 제각각(위아래 안 맞음)이고 여백이 없어 지저분. 종료 옵션도 "0=종료"와 "9=서버 종료"의 차이가 불명확.
- **해결:**
  - **구분선 통일**: 폭 40칸의 `BAR`(`=`)·`DASH`(`-`) 상수로 목록·요청·응답 섹션을 동일 폭으로, 제목은 별도 줄, 섹션마다 한 줄 띄움.
  - **종료 옵션 명확화**: `9) 클라이언트만 종료 (서버는 계속 실행)`, `0) 전체 종료 (서버까지 종료)`로 라벨·동작을 분리. 클라이언트 종료가 서버를 죽이지 않는 실무 관례를 반영하되, 필요 시 `0`으로 서버까지 함께 종료.

## 알려진 한계 (과제 범위상 남겨둔 것)

- **connection-per-thread의 한계:** 연결마다 스레드를 만드므로 수천~수만 동시접속에는 부적합(스레드 생성 비용·메모리). 스레드 생성 실패 자체는 `try/catch`로 방어(#29)했지만, 대규모 성능이 필요하면 **스레드풀**이나 `select`/`epoll`/IOCP 기반 이벤트 다중화가 필요.
- **단순화한 헤더 파싱:** `Content-Length`를 버퍼 전체 문자열 검색으로 찾으므로, 경로 등에 `Content-Length:` 문자열을 심은 악의적 요청은 해당 요청의 본문 길이를 잘못 계산할 수 있음(정상 요청엔 무영향, 멀티스레드라 서버 전체는 안 멈춤). 정식 파서는 헤더를 줄 단위로 파싱해야 함.
- **클라이언트 입력 견고성:** 대화형 클라이언트 메뉴에 숫자가 아닌 값을 넣으면 `cin`이 실패 상태가 됨(대개 깔끔히 종료되나 방어적 처리는 없음).
- **DB 잠금 직렬화:** `buildResponse` 전체를 뮤텍스로 잠가 DB 작업이 직렬화됨(정확성 우선). 읽기 병렬성이 필요하면 스레드별 별도 연결이나 읽기/쓰기 잠금 분리가 필요.
- **slowloris형 공격:** 본문을 아주 느리게 보내며 연결을 오래 점유하는 공격은 크기 상한으로는 못 막음. 본문 읽기 전용 타임아웃이 필요(멀티스레드라 한 연결이 서버 전체를 멈추지는 않지만, 스레드가 계속 쌓일 수 있음).
